# AI Radar：可执行的分阶段开发提示词

> **使用方式：** 先发送“总控指令”，再按顺序发送阶段一到阶段六。必须先完成并验收当前阶段，才能开始下一阶段。

## 0. 总控指令（首次发送）

```text
你是一名资深全栈工程师。请在当前工作目录实现一个可本地运行的 MVP 项目，名称为“AI Radar”。

项目目标：为 AI 内容创作者聚合近期 AI 前沿与热点信息，形成可追溯的话题，并生成可编辑的中文文章草稿、配图提示词和 Markdown 导出文件。

执行规则：
1. 严格按我后续给出的阶段提示执行；一次只完成一个阶段，不跳过验收标准。
2. 先阅读 AGENTS.md、现有代码和配置；遵守既有风格，不删除或覆盖无关文件。
3. 每阶段都直接修改代码、运行相关验证并修复自己引入的问题；不要只给计划、伪代码或代码片段。
4. 每阶段结束后报告：已修改文件、验证结果、下一阶段事项；随后等待下一条提示。
5. 优先实现可演示的最小闭环；不实现账号体系、支付、分布式爬虫、反爬绕过或生产级调度。
6. 所有外部请求必须设置超时并捕获异常；单个数据源失败不能导致应用崩溃。
7. 不绕过登录、验证码、访问控制、robots 规则或平台风控；不下载、复制或再发布原帖图片、视频，只保存链接和短摘要。
8. 每条采集记录必须保存来源名称、原始 URL、作者或频道名（可获取时）、发布时间和抓取时间。

技术约束：Python 3.11+、FastAPI、SQLAlchemy、SQLite、Pydantic Settings、APScheduler；前端采用 Jinja2 + HTMX + Tailwind CSS；界面中文、桌面优先并支持移动端；使用 .env 管理密钥；没有 API Key 时演示模式仍须完整可用；提供 README 和一条本地启动命令。

产品闭环：采集/演示数据 → 热点话题 → 话题详情与证据 → 文章草稿 → 配图提示词 → Markdown 导出。
```

## 1. 阶段一：初始化、配置与演示数据

```text
现在执行阶段一：初始化 AI Radar 项目骨架、数据库和可重复使用的演示数据。

实施内容：
1. 创建清晰的 FastAPI 项目结构，至少分离应用入口、配置、数据库、ORM 模型、业务服务、路由、模板、静态资源和测试目录。
2. 配置 SQLAlchemy + SQLite，应用启动时自动创建数据表。
3. 使用 Pydantic Settings 读取 .env，并提供 .env.example，至少包含 APP_ENV、DATABASE_URL、DEMO_MODE、RSS_URLS、GITHUB_TOKEN、X_BEARER_TOKEN、COMFYUI_URL。
4. 建立数据模型：
   - SourceItem：platform、external_id、title、content、url、author、published_at、fetched_at、metrics_json、language、raw_json；
   - Topic：title、summary、heat_score、freshness、latest_published_at、status；
   - TopicEvidence：topic_id、source_item_id、relevance_score；
   - Draft：topic_id、mode、title、content_markdown、image_prompt、editor_params_json、created_at、updated_at。
5. 实现演示数据初始化：无外部 API、网络或密钥时，也生成多个 AI 来源记录、话题、证据和至少一篇草稿。
6. 建立健康检查接口和首页占位页面，显示应用已启动及演示模式状态；编写 README 的安装与启动初稿。

验收标准：空白 .env 或没有 API Key 时应用可启动且首页能打开；SQLite 中有核心表和演示数据；README 启动命令可用；运行本阶段相关验证并报告结果。
```

## 2. 阶段二：热点仪表盘与话题闭环

```text
现在执行阶段二：完成可操作的热点仪表盘、话题详情和草稿编辑闭环。

实施内容：
1. 首页显示近 24 小时采集条数、热点话题数、已生成草稿数和各数据源状态。
2. 显示热点话题列表；每项包含标题、热度、新鲜度标识、来源平台标签、证据数量、最近发布时间和详情入口。
3. 提供关键词输入框，默认关键词：LLM、agent、reasoning、multimodal、open source、model release、AI、人工智能、大模型、智能体、多模态、开源模型。
4. 创建话题详情页，展示摘要、证据列表，以及每条记录的来源、作者、发布时间、抓取时间、短摘要和原始链接。
5. 创建草稿编辑页：选择写作模式、编辑标题和 Markdown、保存草稿、查看配图提示词、导出 .md 文件。
6. 使用 HTMX 实现局部刷新或提交反馈；按钮具有加载、成功、失败和空状态。不得引入 React/Next.js。
7. 使用简洁的中文资讯产品视觉：信息层级清楚、移动端不溢出，避免巨型标题和无意义装饰。

验收标准：演示模式下可从首页进入详情、编辑草稿、保存并导出 Markdown；页面显示来源链接与元数据；无数据时有明确空状态；完成路由、模板与导出验证。
```

## 3. 阶段三：统一采集框架与公开数据源

```text
现在执行阶段三：实现可扩展的采集框架，并接入不需要登录绕过的公开 AI 信息源。

实施内容：
1. 定义统一采集适配器接口和 SourceItem 标准化逻辑；单个适配器失败时记录可见错误并继续其他适配器。
2. 实现去重：优先 platform + external_id；缺少 external_id 时使用 URL 哈希。重复采集不得生成重复记录。
3. 入库前进行可配置的 AI 相关性过滤，第一版可使用关键词匹配，并保留过滤依据。
4. 接入公开来源：arXiv API（cs.AI、cs.CL、cs.LG 近期论文）、GitHub REST API（近期创建或活跃的 AI 开源项目）、Hacker News Algolia API（AI、LLM、agent、multimodal 讨论）和 .env 中 RSS_URLS 指定的 RSS；可选接入 Hugging Face Daily Papers 的公开 RSS/页面适配器。
5. 用 APScheduler 提供定时采集和受控的“立即采集”入口；页面展示运行中、成功、失败和每个来源状态。
6. 所有请求配置合理超时与 User-Agent；网络不可用时继续使用已有数据或演示数据。

验收标准：“立即采集”可执行并返回可读状态，页面不被阻塞；适配器超时或异常不会造成应用整体失败；至少为去重、标准化或来源失败隔离添加关键测试。
```

## 4. 阶段四：话题聚合与文章生成

```text
现在执行阶段四：实现从来源证据到热点话题、再到可控文章草稿的生成逻辑。

实施内容：
1. 根据关键词、标题相似度或其他可解释规则将 SourceItem 聚合为 Topic；计算可解释的热度和新鲜度，并关联 TopicEvidence。
2. 草稿生成前必须读取当前 Topic 下的证据。信息不足时明确写“现有信息不足以确认”，不得补写细节。
3. 提供四种写作模式：新闻快讯（300–500 字，事实优先）、编辑解读（800–1,200 字，包含论点、证据和局限）、创作者选题（500–800 字，突出受众、角度、争议点和可做内容）、技术拆解（1,000–1,500 字，解释方法、指标、局限和复现线索）。
4. 增加可编辑参数：目标读者、文风、立场强度、目标长度、禁用词、必须出现的事实或来源、必须避免的角度。
5. 如接入 LLM，设计可替换生成 Provider；没有密钥时，必须使用基于证据的确定性模板生成，不能让主流程失效。
6. 草稿必须保留可追溯来源链接。技术名词保留原文，并在首次出现时给出简短中文说明。

写作质量规则：开头直接写最重要的新事实、变化或冲突；避免“近日”“随着……发展”“值得注意的是”等空话。每段只表达一个具体意思，优先写明主体、动作、时间、数字、产品/模型/论文名和来源。仅在有证据时使用“首次”“领先”“爆发”“颠覆”“重大”等强判断。不虚构个人体验、采访、测试结果、业内观点或平台评论，并明确区分事实、推断和未知部分。结尾给出具体观察，避免“总而言之”“未来可期”等套话。

验收标准：可对同一话题生成并保存至少一种可编辑草稿；草稿含来源链接且不虚构来源外事实；没有 LLM 时生成、编辑和导出仍可用。
```

## 5. 阶段五：可选数据源与配图能力

```text
现在执行阶段五：补充可选数据源，并实现不强依赖图像服务的配图能力。

实施内容：
1. B 站只采集公开可访问的数据。优先复用 bilibili-api-python；若不稳定，使用公开热门榜单/API 适配器。不得绕过登录、风控或访问限制。
2. 仅当 X_BEARER_TOKEN 存在时，通过官方 X API v2 获取近期公开帖子；未配置时页面显示“X 数据源未配置”，不报错。
3. 定义 ImageProvider：ComfyUIProvider 仅在 COMFYUI_URL 配置后调用 ComfyUI API；DisabledProvider 在未配置时只生成配图提示词和占位状态，不抛异常。
4. 根据文章主题和写作模式生成适用于图片模型的中文配图提示词；不得下载、复制或展示来源帖的原始图片、视频。
5. 配图任务显示排队、生成中、成功、失败或未配置状态，并安全展示服务返回的生成结果地址（如有）。

验收标准：B 站、X 或 ComfyUI 任一项缺失或失败时，不影响既有功能；未配置图像服务时仍可完成草稿和配图提示词流程；数据源状态在界面中清晰可见。
```

## 6. 阶段六：编辑检查、测试与交付

```text
现在执行阶段六：完善编辑质量检查、测试、文档和最终本地验证。

实施内容：
1. 为草稿增加“编辑检查”面板：事实性陈述数量、已关联来源段落比例、无关联来源的判断句、重复句式或高频连接词、可能空泛的句子、需人工复核的强结论。
2. 提供一键改写选项：“更具体”“更克制”“更像新闻编辑”“更像技术作者”。改写不得删除来源链接或添加无来源事实。
3. 补充核心单元/集成测试：演示模式启动、去重、来源异常隔离、话题聚合、草稿生成、Markdown 导出。
4. 完善 README：项目用途、架构、环境变量、启动命令、演示模式、数据源限制与合规边界、测试命令和已知限制。
5. 实际运行应用并验证：采集/演示数据 → 热点 → 详情与证据 → 草稿 → 配图提示词 → Markdown 导出。
6. 修复本阶段发现且由本项目引入的明显问题；不要通过删除功能或弱化验收要求来通过验证。

最终验收标准：使用 README 命令可在本地启动并在浏览器打开；没有 API Key 时演示模式完整展示闭环；配置可用数据源后可真实拉取公开内容；来源错误可见但不会导致应用崩溃；最终报告列出启动命令、测试命令、关键文件、已实现功能和已知限制。
```

## 7. 发送顺序

1. 首次将“总控指令”发送给 Codex。
2. 再依次发送“阶段一”到“阶段六”，每次只发送一个阶段。
3. 如果 Codex 报告当前阶段未通过验收，继续要求它修复当前阶段，直到通过。
4. 所有阶段完成后，要求进行一次最终回归验证，不新增无关功能。
